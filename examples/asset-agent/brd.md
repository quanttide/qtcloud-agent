# 资产智能体协作上下文优化方案 — 商业需求文档

文档状态：v1.0 草案
角色：产品经理
日期：2026-05-16

—

1. 商业背景与问题

1.1 背景

量潮平台资产库包含多个领域（平台、示例、库），每个领域含 docs、src、test、config 等子目录。当前使用 LLM 智能体（Opencode）辅助资产维护。

随着资产库膨胀，单次智能体调用需要加载的上下文越来越大——整个仓库的结构、所有相关文件、历史记录。LLM 上下文窗口有限，全量加载既不经济也不有效。

1.2 问题本质

传统思路是在"不拆项目"的前提下做上下文裁剪——这需要复杂的排序、预算、截断逻辑，且无论如何裁剪都有丢失关键信息的风险。

核心洞察：**上下文不是"裁"出来的，是"拆"出来的。**

与其在单个大项目中让每个智能体都面对全量上下文，不如：
1. 将大项目拆解为若干小项目，每个小项目只包含一个智能体所需的最小上下文
2. 智能体间通过结构化事件交换必要信息，跨域上下文按需传递而非全量加载

—

2. 方案概述

2.1 核心机制

```
                    ┌──────────────┐
                    │   元仓库      │
                    │  (meta-agent) │
                    └──────┬───────┘
                           │ 事件（策略/约束）
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ 单仓 A   │ │ 单仓 B   │ │ 单仓 C   │
        │ plat-ag  │ │ test-ag  │ │ lib-ag   │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │ 事件         │ 事件        │ 事件
        ┌────┴────┐   ┌────┴────┐  ┌────┴────┐
        │doc-agent│   │doc-agent│  │doc-agent│
        │code-ag  │   │code-ag  │  │code-ag  │
        │test-ag  │   │test-ag  │  │test-ag  │
        └─────────┘   └─────────┘  └─────────┘
```

2.2 上下文优化原理

| 策略 | 说明 |
|------|------|
| 按项目拆分 | 每个单仓=独立小项目，智能体只需加载自己所在小项目的文件 |
| 按层级缩小视野 | meta-agent 只关心元仓库配置，不关心单仓内部实现细节 |
| 事件传递上下文 | 跨域信息通过事件 payload 传递，接收方无需重新加载发送方的全量文件 |
| 事件即上下文 | 事件的 data 字段携带足够信息，下游智能体无需回溯源文件 |

2.3 对比

| 维度 | 传统做法（不拆） | 本方案（拆+事件） |
|------|-----------------|-------------------|
| 单次上下文量 | 50K+ token（全库） | 5-10K token（仅小项目） |
| 跨域信息获取 | 全量加载其他域文件 | 事件携带必要信息 |
| 上下文复杂度 | 多域混杂，排序困难 | 单域内聚，天然有序 |
| 扩展性 | 项目膨胀→上下文继续膨胀 | 新项目=新智能体，上下文规模恒定 |

3. 上下文交换规格

3.1 事件即上下文接口

智能体间通信采用统一事件格式（CloudEvents 风格），事件本身就是上下文传递的载体。

```json
{
  "id": "evt-20260516-001",
  "type": "drd.finalized",
  "source": "/平台/docs/drd",
  "subject": "支付设计.md",
  "data": {
    "status": "final",
    "summary": "支付服务采用策略模式，支持微信/支付宝/银联",
    "key_decisions": ["选用策略模式", "异步对账", "T+1 结算"],
    "changed_files": ["支付设计.md", "架构决策.md"],
    "review_required": ["qa", "code"]
  }
}
```

下游智能体收到此事件后，无需重新加载 `支付设计.md` 全文——事件的 `data.summary` 和 `data.key_decisions` 已携带核心上下文。

3.2 事件类型作为上下文标识

事件 type 告知接收方需要加载哪些额外上下文：

| 事件类型 | 接收方 | 需要补充加载的上下文 |
|----------|--------|---------------------|
| drd.finalized | qa-agent | 仅加载 DRD 约束章节 + 事件 data |
| drd.finalized | code-agent | 仅加载 DRD 接口定义章节 + 事件 data |
| design.change.request | drd-agent | 仅加载 QA 反馈文件 + 原 DRD 文件 |
| test.passed | code-agent | 仅加载测试报告摘要（事件 data 中） |

3.3 小项目上下文边界

每个小项目的上下文范围：

```
小项目根/
├── README.md              → 项目概述（必须加载）
├── docs/                  → 该智能体相关的文档（按需加载）
│   └── 当前任务文件.md
├── config/                → 该智能体配置
│   └── rules.yaml
└── .events/               → 近期相关事件（参考上下文）
    └── latest.json
```

不加载：其他小项目的文件、元仓库级策略细节、历史归档。

4. 商业目标

目标 1：单次智能体调用上下文 ≤ 10K token
衡量方式：所有智能体调用日志中的 token 计数。

目标 2：智能体输出中 > 90% 的引用信息来自其小项目内文件或事件 data
衡量方式：reviewer 智能体抽样审计引用溯源。

目标 3：新项目接入成本 ≤ 30 分钟
衡量方式：从新建仓库到配置好智能体并跑通首次协作的时间。

目标 4：无需手动裁剪上下文
衡量方式：策略完全由拆项目 + 事件设计保证，用户不需写任何上下文裁剪规则。

—

5. 功能需求

FR-01 项目拆分标准：定义元仓库、单仓、分层的目录结构和智能体分配规范。

FR-02 事件上下文规范：定义各类型事件的 data 字段应包含哪些摘要信息，确保事件自身携带足够上下文。

FR-03 事件触发路由：事件按规则路由到目标智能体，事件 data 作为上下文的一部分传递给接收方。

FR-04 小项目上下文自包含：每个小项目的文件范围即为智能体的上下文范围，无需额外配置。

FR-05 事件上下文审计：记录每次事件传递的 data 大小和接收方实际使用的上下文来源分布。

—

6. 约束与依赖

· 依赖现有资产目录树结构（元仓库 → 单仓 → 分层），项目拆分须与目录结构对齐
· 依赖事件格式规范（CloudEvents 风格），data 字段设计决定上下文质量
· 小项目的粒度由人工定义，系统不自动拆分

—

7. 成功指标

| 指标 | 当前（模拟大项目） | 目标（拆小项目+事件） |
|------|-------------------|---------------------|
| 单次上下文量 | 50K+ token | ≤ 10K token |
| 跨域文件加载数 | 10-30 个/次 | 0-2 个/次（从事件 data 获取） |
| 手动裁剪规则数 | 需要（YAML 排除列表） | 0 |
| 新智能体接入时间 | 需理解全库 > 2h | 理解小项目 ≤ 30min |
